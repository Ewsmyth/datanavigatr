Boolean Logic:

A system of algebraic operations that uses true/false values to define relationships between different conditions. Allowable boolean operators are: AND, OR, NOT. The AND operators requires all specified conditions must be true for a record to be included in the results. The OR operator requires at least one of the specified conditions must be true (this could also be both/all conditions). The NOT operator excludes records that meet the specified condition.

Condition:

A condition is a specified relationship between a Data Field Name and a Data Value defined by an Operator. An example of a condition is: {{ "TX_USER_ID==(user1)" | e }}

CONTENT:

The content of an online interaction.

CONTENT_ID:

A unique numerical identifier associated to a specific record of user uploaded data.

Creating a Mission:

Create a mission by selecting the 3 vertical dots icon on the top right of the screen and then select "Mission". Assign a name to our mission, provide a brief explanation/justification, and then share the mission with other users. When creating queries you can select which mission to save it under and all members of that mission can view that query.

Data Record:

A single set of related Data Fields within a database, representing a unique entry.

Data Value:

Specific content stored within a Data Field for a given Record

Dark Mode:

To change from light mode to dark and back select the 3 vertical dots on the top right then select "Settings", you can then select the theme you prefer.

Deleting a Saved Machine:

To delete a saved machine you can right-click on the machine you wish to delete then select delete.

EVENT:

The type of event for a Data Record.

Exporting Query Results:

Query results can be exported to a .xlsx file by clicking the Export on the top right. The Export function will only export the result that are currently displayed on the page and in the format of the page.

FINANCE_ID:

User submitted banking information.

GROUP_CHAT_NAME:

A user assigned name of a group chat in Reporter.

GROUP_PARTICIPANTS:

A list of user IDs associated to a group chat in Reporter.

id:

The unique identifier for a Data Record.

Ingest:

As of the current version of DataNaviGatr ingestion only works with Reporter databases. To ingest data from a Reporter database you need to navigate to the ingest page by click on the 3 vertical dots on the top right of the home page then select "Ingest". The Ingest page allows you to SSH into the machine that is running Reporter and export the database into QDB1. You can also save your machine's profile for quick and easy ingest operations.

INITIAL_TIME:

The time a Data Record was first submitted to the database, if no value refer to the INITIAL_TIME.

LOCATION:

A location specified by user submitted data, not a location request from the device itself.

Machine Name:

The Machine Name input is a user preference input, it's only purpose is allow you to assign names to machines for easy reference. The machine name is not used in the data ingestion process, so you can make it what ever you want.

MEDIA:

The file name of upload images and videos. Future updates will display the actual content.

Operator:

Values used to define the relationship between different types of data. Allowable operators are: == (equals), = (contains), != (does not equal), < (less than), > (greater than).

Path to .db File:

The file path to the .db (SQLite database file type) file of the website being ingested. All of the sites this system is meant to work with store their .db file in the Docker volumes directory (/var/lib/docker/volumes/), each database volume is typically named after the website followed by -data (I.E. reporter-data). Docker volumes also create another sub-directory name "_data", that is where the .db files should be if the sites were set up correctly. An example of the entire input could look like this: "/var/lib/docker/volumes/reporter-data/_data/reporter-data.db" (ps: The quotations are not part of the file path).

QDB1:

QDB1 or Queryable Database 1 is where all ingested data is stored and queried. The purpose of QDB1 is to translate different databases into one uniform system that can be easily referenced.

Query Parameters:

The query parameters is a structured grouping of Conditions and Boolean Operators. The proper query structure is condition {{ "" | e }} boolean operator {{ "" | e }} condition etc... An example could look like this: {{ "TX_USER_ID==(user1)" | e }} AND CONTENT=(hello)

Query View:

When veiwing query results you can re-arange the table by left-click-hold-and-drag.

RX_USER_ID:

A unique numerical identifier assigned by the server for a user's account that is not visable to the user.

Saving a Machine:

To save a machine in the ingest page, enter the parameters of the machine you want to ingest data from then select the "Save Parameters" button on the bottom left.

Server IP:

Internet Protocol (IP) address of the machine that is running the website in which you want ingest data from.

SQL Query:

The SQL Query that reformats the ingested data into QDB1's format. You can create your own SQL Query by creating a new ingestion query but you should not have to.

SSH Password:

The password for the SSH Username account.

SSH Username:

The SSH Username input is the username for the machine that you are attempting to pull the data from. Because most ingestion for the sites that this system is designed for are Docker containers it is easiest to enable root login and root SSH on the machine running the sites; this allows you to ingest data with the root user and avoid issues with not having access to the proper directories.

TX_ACCT_AUTH:

The authority of a user's account assigned by administrative users.

TX_ACCT_PASSWD:

Password assigned by the user and hashed by the server.

TX_ACCT_STATUS:

A true/false value that determines a user's account status. 1 typically meaning active and 0 typically meaning de-activated.

TX_ASSOC_EMAIL:

Unique identifier assigned by the user to authenticate with the server, is visible to the user.

TX_ASSOC_FNAME:

The firstname associated to a user's account, this is assigned by the user and is not unique.

TX_ASSOC_LNAME:

The lastname associated to a user's account, this is assigned by the user and is not unique.

TX_ASSOC_PHONE:

A user assigned phone number associated to a user's account.

TX_ASSOC_TEAM:

A group of online users to which a specific TX_USER_ID is a part of.

TX_ASSOC_USRNM:

Unique identifier assigned by the user to authenticate with the server, is visable to the user.

TX_CTL_BY:

A unique numerical identifier of a user that manages the account of the given TX_USER_ID in a specific Data Record.

TX_USER_ID:

A unique numerical identifier assigned by the server for a user's account that is not visable to the user.

UP_TIME:

The time the event took place or the most recent time of Data Record update.
